# 跨阶段质量审计框架

## 何时使用

- 任何阶段的产出需要审查
- 需要生成标准化审计报告
- 用户讨论审计、审查、质量检查

## 核心原则

**审计是硬性要求，不可跳过。** 每个阶段的输出必须通过AI审计后才能进入人类审查。

### 审计Agent行为准则

1. **独立审查**：不依赖生成Agent的自我评估
2. **不生成内容**：只做质量检查，不生成新内容
3. **结构化报告**：输出标准化的审计报告
4. **分类问题**：🔴严重 / 🟡一般 / 🔵建议

## 统一审计报告格式

```markdown
# 审计报告 — [文档/代码名称]

## 📋 基本信息
- 审计对象: [文档/代码名称]
- 审计时间: [时间戳]
- 审计Agent: [Agent名称]
- 审计版本: [版本号]

## ✅ 通过项
- [列出无明显问题的方面]

## ⚠️ 警告项
- [列出需要注意但不阻塞的问题]
  - 建议: [具体建议]

## ❌ 不合格项
- [列出必须修改的问题]
  - 原因: [为什么不合格]
  - 建议修改: [具体修改建议]

## 📝 综合修改建议
1. [建议1]
2. [建议2]
3. [建议3]

## 📊 综合评分
- 完整性: X/10
- 一致性: X/10
- 可行性: X/10
- 规范性: X/10
- 安全性: X/10（如适用）
- 总分: X/50

## 🚦 审计结论
- [ ] **通过** — 可以直接进入下一阶段
- [ ] **有条件通过** — 需处理⚠️警告项后进入下一阶段
- [ ] **不通过** — 需修复所有❌不合格项后重新审计
```

## 各阶段审计维度

### Phase 0: 项目初始化审计

| 维度 | 检查项 |
|------|--------|
| Agent配置 | SOUL.md是否定义完整？能力边界是否清晰？ |
| 协作规范 | AGENTS.md是否定义命名/风格/规模/文档规范？ |
| 安全规范 | 是否禁止危险操作？是否定义输入验证/日志脱敏？ |
| 项目隔离 | CLAUDE.md是否与全局配置隔离？ |
| 目录结构 | 是否包含audit/、prompts/、prototypes/目录？ |

### Phase 1: 需求工程审计

| 维度 | 检查项 |
|------|--------|
| 完整性 | 是否覆盖所有用户角色？是否遗漏非功能性需求？ |
| 一致性 | 功能需求与约束条件是否矛盾？ |
| 可行性 | 约束条件是否现实可行？ |
| 风险识别 | 是否存在技术/合规/资源风险？ |
| 优先级 | 是否区分MVP和后续迭代？ |

### Phase 2: 架构设计审计

| 维度 | 检查项 |
|------|--------|
| 架构完整性 | 是否覆盖所有功能需求？模块划分是否合理？ |
| 技术选型 | 是否成熟稳定？是否有更好替代方案？ |
| 性能设计 | 是否满足性能要求？缓存/索引策略是否合理？ |
| 安全设计 | 认证/授权/加密/审计是否完善？ |
| 可扩展性 | 是否考虑用户/数据量增长？ |
| 可靠性 | 容错/监控/备份/恢复方案是否完善？ |

### Phase 2.5: 功能设计审计

| 维度 | 检查项 |
|------|--------|
| 权限设计 | 是否覆盖所有角色？最小权限原则？ |
| 工作流设计 | 状态机是否闭环？驳回路径是否完整？ |
| UI设计 | 设计令牌是否覆盖所有视觉元素？组件是否可复用？ |
| 原型 | 是否覆盖所有核心页面？交互是否自然？ |

### Phase 3-4: 编码实施审计

| 维度 | 检查项 |
|------|--------|
| 安全性 | SQL注入/XSS/硬编码密钥/eval()？ |
| 逻辑正确性 | 边界条件/异常处理/返回值类型？ |
| 规范性 | 编码规范/函数长度/类型注解/中文注释/文档同步？ |
| 测试覆盖 | 正常/边界/异常场景？ |
| 架构一致性 | 是否引入架构外库？模块职责是否符合设计？ |

### Phase 5: 测试工程审计

| 维度 | 检查项 |
|------|--------|
| 覆盖度 | 是否覆盖所有输入字段和业务规则？ |
| 边界完整性 | 是否覆盖所有边界值？ |
| 异常充分性 | 是否覆盖常见异常输入和安全攻击？ |
| 业务场景 | 是否覆盖所有用户角色和权限组合？ |

### Phase 6: 文档交付审计

| 维度 | 检查项 |
|------|--------|
| 完整性 | 是否覆盖所有功能模块？部署步骤是否完整？ |
| 准确性 | 文档是否与代码一致？配置参数是否正确？ |
| 可读性 | 语言是否清晰？是否有足够示例？ |
| 一致性 | 文档风格/术语/版本号是否一致？ |

### Phase 7: 变更管理审计

| 维度 | 检查项 |
|------|--------|
| 变更分析 | 等级判断是否正确？影响范围是否完整？ |
| 同步策略 | 文档/测试是否同步更新？ |
| 风险评估 | 风险点是否识别？回退方案是否可行？ |

### Phase 8: 版本管理审计

| 维度 | 检查项 |
|------|--------|
| 修复有效性 | 补丁是否真正修复问题？回归测试是否通过？ |
| 副作用分析 | 是否影响其他功能？TDAD测试是否通过？ |
| 安全性 | 是否引入新漏洞？是否有敏感信息泄露？ |
| 规范性 | 代码规范/文档更新/提交信息？ |

## 评分标准

### 单项评分（0-10分）

| 分数 | 说明 |
|------|------|
| 9-10 | 优秀，无明显问题 |
| 7-8 | 良好，有少量警告 |
| 5-6 | 一般，有多个警告或少量不合格 |
| 3-4 | 较差，有多个不合格项 |
| 0-2 | 严重问题，需要重写 |

### 总分判定

| 总分 | 结论 |
|------|------|
| 40-50 | **通过** — 可以直接进入下一阶段 |
| 30-39 | **有条件通过** — 需处理⚠️警告项 |
| < 30 | **不通过** — 需修复所有❌不合格项后重新审计 |

## 审计触发条件

### 自动触发

- 每个阶段生成初稿后
- 每次迭代修改后
- 每次代码提交前

### 手动触发

- 人类要求审查特定内容
- 发现问题需要深入分析
- 争议决策需要第三方评估

## 审计执行步骤

1. **读取审计对象**：完整读取需要审查的文档/代码
2. **读取参考文档**：读取相关的规范/设计/需求文档
3. **逐项检查**：按审计维度逐项检查
4. **记录问题**：分类记录通过项/警告项/不合格项
5. **生成评分**：按评分标准打分
6. **输出报告**：按统一格式输出审计报告
7. **等待反馈**：等待生成Agent修改或人类审批

## 注意事项

- **不生成内容**：审计Agent只审查，不生成新内容
- **独立判断**：不依赖生成Agent的自我评估
- **客观公正**：基于事实和标准，不主观臆断
- **具体建议**：每个问题都要有具体的修改建议
- **可追溯**：审计结论要有依据，引用相关规范
- **及时性**：审计应在生成后尽快执行，不延迟流程
