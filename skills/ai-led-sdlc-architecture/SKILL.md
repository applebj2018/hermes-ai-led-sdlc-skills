# Phase 2: 架构设计

## 何时使用

- 需求文档(INITIAL.md)已批准，进入架构设计阶段
- 需要设计系统架构、模块划分、接口协议
- 需要进行技术选型
- 需要设计数据模型

## 核心原则

### HARD-GATE（阶段入口/出口）

```
< HARD-GATE: Phase 2 入口 >
- 前置检查：确认 INITIAL.md 已存在且状态为 approved
- 禁止跳过：即使项目简单，架构文档可以简短但不可省略
- 未批准的 INITIAL.md 不得进入 Phase 2
</ HARD-GATE >

< HARD-GATE: Phase 2 出口 >
- 禁止跳过审计：ARCHITECTURE.md 和 DATA-MODEL.md 必须经 AI 审计
- 禁止跳过人类审批：审计通过后必须等待用户明确批准
- 禁止直接进入 Phase 2.5/4：架构未批准前，不得生成功能设计或代码
</ HARD-GATE >
```

### Anti-Pattern（常见错误模式）

**"架构太复杂了，直接编码吧"** — 没有架构设计的编码是盲人摸象。模块边界不清、接口协议缺失、数据模型混乱，后期重构成本指数级增长。

**"技术栈随便选"** — 不搜索业界最佳实践的技术选型是赌博。至少对比 2-3 个方案，考虑成熟度、兼容性、性能、安全性。

**"过度设计"** — 为不确定的未来设计架构是浪费。架构服务于当前需求，保持简洁。

### 简洁优先

- **功能决定形式**：不追求复杂架构，架构服务于功能
- **成熟技术栈**：优先选择成熟、稳定、社区活跃的技术
- **最少依赖**：每个依赖必须有明确理由
- **模块化设计**：高内聚、低耦合

### 技术选型决策框架

| 维度 | 评估标准 |
|------|---------|
| **成熟度** | 是否有稳定版本？社区活跃度？ |
| **兼容性** | 是否与现有技术栈兼容？ |
| **性能** | 是否满足性能要求？ |
| **安全性** | 是否有已知安全漏洞？更新频率？ |
| **可维护性** | 文档是否完善？学习曲线？ |
| **成本** | 开源/商业许可？运行成本？ |

### UI设计原则

- **现代感**：简洁、留白、层次分明
- **动感丰富**：微动画、过渡效果、反馈及时
- **体验好**：操作流畅、错误提示友好、加载状态明确
- **响应式**：适配不同屏幕尺寸
- **无障碍**：WCAG 2.1 AA标准

## 技术栈推荐矩阵

**注意**：本矩阵提供**业界通用推荐**，列出多种选项供选择。用户的**默认偏好**见 `ai-led-sdlc-user-preference` 技能。

| 场景 | 推荐方案 | 备选方案 |
|------|---------|---------|
| **后端API** | Python FastAPI / Node.js Express | Go Gin, Java Spring Boot |
| **前端框架** | React + TypeScript / Vue 3 | Svelte, Angular |
| **数据库** | PostgreSQL | MySQL, SQLite |
| **缓存** | Redis | Memcached |
| **消息队列** | RabbitMQ / Redis Streams | Kafka, Celery |
| **搜索引擎** | Elasticsearch | Meilisearch, Typesense |
| **向量数据库** | Milvus / Qdrant | Pinecone, Weaviate |
| **对象存储** | MinIO / AWS S3 | 阿里云OSS, 腾讯云COS |
| **ORM** | SQLAlchemy / Prisma | Django ORM, TypeORM |
| **测试框架** | pytest / Jest | unittest, Vitest |
| **API文档** | Swagger/OpenAPI | Redoc, Stoplight |

**技术选型流程**：
1. 加载 `ai-led-sdlc-user-preference` 获取用户默认偏好
2. 加载 `ai-led-sdlc-domain-spec` 获取行业特定约束
3. 根据需求从推荐矩阵中选择技术
4. 如有覆盖，在 AGENTS.md 中明确声明

**详细说明**：见 `references/tech-stack-decision-framework.md`

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | 架构是否满足所有功能需求 | 无遗漏模块 |
| 2 | 技术选型是否合理 | 成熟、兼容、性能达标 |
| 3 | 性能设计是否可行 | 满足INITIAL.md要求 |
| 4 | 安全设计是否完善 | 认证/授权/加密/审计齐全 |
| 5 | 数据模型是否合理 | 范式合理、索引完善 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **读取输入**：读取已批准的 INITIAL.md 和 AGENTS.md
2. **外部搜索**：搜索同类系统的架构设计最佳实践
3. **技术选型**：根据需求和技术栈推荐矩阵选择技术
4. **架构设计**：生成 ARCHITECTURE.md 初稿
5. **数据建模**：生成 DATA-MODEL.md 初稿
6. **提交审计**：将设计文档提交给审计Agent审查
7. **迭代修改**：根据审计报告修改设计
8. **人类审批**：按审批清单逐项检查
9. **版本化**：将最终版纳入版本管理
10. **移交**：输出作为 Phase 2.5 的输入

## 注意事项

- **不要过度设计**：架构服务于当前需求，不为不确定的未来设计
- **技术债意识**：记录每个技术选型的权衡和妥协
- **文档先行**：架构文档在编码前必须完成
- **性能前置**：在架构阶段就考虑性能，不要等到编码后优化
- **安全前置**：安全是架构属性，不是补丁
- **提醒用户**：如果技术选型有重大影响，主动提醒用户确认
- **避免重复**：技术栈推荐矩阵提供业界通用选项，用户的默认偏好见 `ai-led-sdlc-user-preference`，行业特定约束见 `ai-led-sdlc-domain-spec`。不要在架构技能中硬编码用户偏好，避免多处维护同一配置。
- **避免重复**：技术栈推荐矩阵提供**通用选项**，用户默认偏好见 `ai-led-sdlc-user-preference`，行业约束见 `ai-led-sdlc-domain-spec`。不要在这些技能之间复制相同内容。
