# Phase 7: 变更管理与迭代

## 何时使用

- 项目已有代码和文档，需要变更
- 需要分析变更影响范围
- 需要决定变更执行路径
- 用户讨论变更、迭代、需求修改

## 核心原则

### HARD-GATE（变更入口/出口）

```
< HARD-GATE: 变更入口 >
- 禁止直接改代码：每次变更前必须先执行变更分析（确定 L1/L2/L3 等级）
- 禁止跳过版本保存：变更前必须先提交当前版本或创建 Git checkpoint
- 禁止未分析就实施：不明确影响范围的变更不得执行
</ HARD-GATE >

< HARD-GATE: 变更出口 >
- L1 变更：必须运行受影响模块的测试
- L2 变更：必须同步更新代码+文档+测试，并运行全量回归
- L3 变更：必须走完整循环（Phase 0→1→2），不得直接改代码
- 禁止跳过回退方案：L2 以上变更必须有明确的回退步骤
</ HARD-GATE >
```

### Anti-Pattern（常见错误模式）

**"小改动不用分析"** — AI 改代码时最大的风险是"优化"掉已有功能。即使是 L1 微小变更，也必须确认影响范围。

**"先改再说，有问题再回退"** — 没有版本保存的变更是赌博。AI 可能把已经做好的功能改没有，回退成本极高。

**"L3 变更当 L1 处理"** — 架构变更走 L1 路径是灾难。技术选型变更必须重新评估需求、架构、数据模型。

**基本原则**：每次做变更前，对现有版本一定要做一次版本保存，因为AI Agent可能把你已经做好的功能改没有！

对于AI Agent，变更成本非常高——它要读所有相关的文档和代码，几乎和重新生成一样高成本。但如果你很清楚代码和文档，找到要改的文件，告诉它，会大大降低修改任务的复杂度。

## 变更分类

### 判断标准：变更是否跨越了"约束锚点"？

| 跨越锚点 | 变更等级 | 同步策略 |
|---------|---------|---------|
| 否（只改实现） | L1 微小变更 | AI直接改代码 |
| 部分跨越（改接口/模块） | L2 局部变更 | AI同时改代码+文档+测试 |
| 全部跨越（改架构） | L3 架构变更 | 完整循环（Phase 0→1→2） |

### 变更等级定义

| 等级 | 定义 | 影响范围 | 典型场景 |
|------|------|---------|---------|
| **L1 微小** | 不涉及架构/接口/数据模型 | 单个函数/参数/样式 | 文案修改、参数调整、UI美化、Bug修复 |
| **L2 局部** | 影响1-2个模块的接口或数据模型 | 模块级 | 增加API字段、修改搜索策略、增加新功能模块 |
| **L3 架构** | 影响核心架构或技术选型 | 系统级 | 更换数据库、增加新引擎、修改核心架构 |

### 实操口诀

- 只改代码内部逻辑 → L1
- 改了接口或数据模型 → L2
- 改了技术选型或核心架构 → L3

## 变更决策流程

```
收到变更需求
    ↓
【AI变更分析】（自动）
    ↓
    是否只改代码内部逻辑？
    ├─ 是 → L1 微小变更 → AI直接改代码 → 完成
    └─ 否 ↓
    是否改了接口或数据模型？
    ├─ 是 → L2 局部变更 → AI同时改代码+文档+测试 → 完成
    └─ 否 ↓
    是否改了技术选型或核心架构？
    ├─ 是 → L3 架构变更 → 完整循环（Phase 0→1→2）
    └─ 否 → 重新分析
```

## 变更分析提示词

```text
# 角色
你是一名变更影响分析专家。

# 任务
分析以下变更需求，确定变更等级（L1/L2/L3）和同步策略。

# 输入
- 变更描述: [用户提出的变更]
- 当前代码结构: [代码目录树]
- 约束锚点: AGENTS.md, 命名规范

# 输出格式

## 变更分析
- 变更等级: L1/L2/L3
- 影响文件: [受影响的文件列表]
- 需要同步的文档: [需要更新的文档列表]
- 需要同步的测试: [需要更新或新增的测试]

## 同步策略
- [具体的同步步骤]

## 风险评估
- [可能的风险点]
- [回退方案]
```

```text
# 角色
你是一名资深工程师，负责实施变更并确保代码、文档、测试同步。

# 任务
实施以下变更，同时更新相关文档和测试。

# 输入
- 变更描述: [用户提出的变更]
- 变更分析: [AI变更分析的输出]
- 约束锚点: AGENTS.md, 命名规范

# 输出要求
1. 修改代码（遵循命名规范、代码风格）
2. 更新文档（DESIGN.md、同名.md文档）
3. 更新测试（覆盖变更点）
4. 运行测试确认通过

# 验证清单
- [ ] 代码变更完成
- [ ] 文档同步完成
- [ ] 测试通过
- [ ] 架构约束未违反
- [ ] 命名规范遵循
```

## 常见变更场景速查表

| 场景 | 变更等级 | 同步策略 | 提示词要点 |
|------|---------|---------|-----------|
| UI样式美化 | L1 | 只改前端 | 只修改CSS/HTML，不修改JS逻辑 |
| 文案修改 | L1 | 只改前端 | 只修改文本内容，不修改结构 |
| 参数调整 | L1 | 只改代码 | 只修改配置参数，不修改逻辑 |
| Bug修复 | L1 | 只改代码 | 修复特定问题，不修改其他功能 |
| 增加API字段 | L2 | 代码+文档+测试 | 同时修改前端、后端、DESIGN.md |
| 修改搜索策略 | L2 | 代码+文档+测试 | 同时修改服务层、DESIGN.md、测试 |
| 增加新功能模块 | L2 | 代码+文档+测试 | 同时修改前端、后端、DESIGN.md |
| 更换技术选型 | L3 | 完整循环 | 更新INITIAL.md、AGENTS.md、重新设计 |
| 修改数据库方案 | L3 | 完整循环 | 更新DATA-MODEL.md、重新设计 |
| 修改核心架构 | L3 | 完整循环 | 更新ARCHITECTURE.md、重新设计 |

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | 变更分析是否准确 | 等级判断正确、影响范围完整 |
| 2 | 文档同步是否完成 | 所有需要更新的文档已更新 |
| 3 | 测试是否通过 | 受影响测试全部通过 |
| 4 | 回退方案是否可行 | 有明确的回退步骤 |
| 5 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **版本保存**：变更前先提交当前版本
2. **变更分析**：AI分析变更等级和影响范围
3. **人类确认**：确认变更方案可行
4. **实施变更**：AI按等级执行同步策略
5. **测试验证**：运行受影响测试
6. **审计审查**：提交给审计Agent审查
7. **人类审批**：按审批清单逐项检查
8. **版本提交**：将变更提交到版本管理

## 注意事项

- **变更前必须保存版本**：这是铁律，AI可能改坏已有功能
- **明确告诉AI改什么文件**：降低AI的搜索成本
- **小步验证**：每完成一个变更点立即验证
- **文档同步**：L2以上变更必须同步更新文档
- **测试覆盖**：变更必须覆盖到测试
- **回退方案**：L2以上变更必须有明确的回退方案
