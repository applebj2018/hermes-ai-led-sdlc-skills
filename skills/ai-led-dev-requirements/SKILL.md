# Phase 1: 需求工程

## 何时使用

- 启动新项目的需求分析阶段
- 需要将模糊的业务想法转化为结构化需求文档
- 需要生成 INITIAL.md 项目规格说明书
- 用户讨论需求、功能、用户角色、业务规则

## 核心原则

### 强制外部搜索

在生成需求文档之前，**必须**先搜索互联网：
1. 参考业界类似系统的最佳实践
2. 了解目标领域的行业标准
3. 发现可能被遗漏的功能点
4. 验证技术选型的可行性

### 强制 Brainstorming

在形成需求文档之前，**必须**进行头脑风暴：
1. 从多个用户角色视角思考需求
2. 考虑正常路径和异常路径
3. 识别边界条件和约束
4. 发现潜在的风险和依赖

## 输入信息收集

向用户确认以下信息（用提问方式引导补充）：

| 类别 | 问题 |
|------|------|
| **项目定位** | 项目的核心目标是什么？解决什么问题？ |
| **目标用户** | 谁会使用这个系统？他们的角色和痛点是什么？ |
| **核心功能** | 系统必须有哪些功能？哪些是MVP，哪些可以后续迭代？ |
| **数据来源** | 系统需要处理哪些类型的数据？数据来源是什么？ |
| **约束条件** | 有哪些技术约束、安全约束、合规约束？ |
| **性能要求** | 预期的用户规模、并发量、响应时间要求？ |
| **集成需求** | 需要与哪些现有系统集成？ |
| **部署环境** | 本地部署、云端部署、还是混合部署？ |
| **预算限制** | 是否有成本限制影响技术选型？ |
| **时间要求** | 是否有明确的交付时间线？ |

## 生成提示词模板

```text
# 角色
你是一名资深技术产品经理，擅长将模糊的业务需求转化为结构化的项目规格说明。

# 任务
基于以下业务需求，生成一份完整的 INITIAL.md 文档。

**请先搜索互联网**，参考业界类似系统的最佳实践，补充功能点和技术建议。

**业务需求描述**：
[用户提供的业务需求]

**目标用户**：
[用户角色列表]

**核心功能**：
[功能列表]

**约束条件**：
[技术/安全/合规约束]

# 输出要求

## INITIAL.md 结构

# [项目名称] — 项目规格说明书

## 1. 项目概述
- 项目背景
- 核心目标
- 成功标准（可量化的验收指标）

## 2. 目标用户
| 角色 | 职责 | 核心需求 | 使用频率 |
|------|------|---------|---------|

## 3. 功能需求

### 3.1 MVP功能（必须实现）
| 功能 | 描述 | 优先级 | 验收标准 |
|------|------|--------|---------|

### 3.2 后续迭代功能
| 功能 | 描述 | 优先级 | 预计版本 |
|------|------|--------|---------|

## 4. 非功能性需求
- 性能要求（响应时间、并发量、吞吐量）
- 安全要求（认证、授权、数据加密、合规）
- 可用性要求（SLA、故障恢复）
- 可维护性要求（代码规范、文档要求）

## 5. 数据来源与集成
| 数据源 | 类型 | 格式 | 集成方式 | 频率 |
|--------|------|------|---------|------|

## 6. 技术约束
- 部署环境
- 技术栈限制
- 合规要求
- 其他约束

## 7. 系统边界
- 做什么（In Scope）
- 不做什么（Out of Scope）

## 8. 风险识别
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|

## 9. 里程碑计划
| 阶段 | 交付物 | 预计时间 |
|------|--------|---------|
```

## Brainstorming 引导模板

在搜索最佳实践后，进行结构化头脑风暴：

```text
# Brainstorming 阶段

## 1. 用户角色视角
从每个用户角色的角度思考：
- [角色A] 每天如何使用系统？核心操作是什么？
- [角色B] 什么场景下会使用系统？痛点是什么？
- ...

## 2. 功能完整性检查
- 核心业务流程是否完整？（创建→编辑→审批→发布→归档）
- 是否有权限管理需求？
- 是否需要版本控制？
- 是否需要审计日志？
- 是否需要数据导出？
- 是否需要通知机制？

## 3. 边界条件识别
- 数据量边界（最大/最小）
- 并发量边界（峰值/谷值）
- 网络边界（离线/弱网）
- 权限边界（越权访问）

## 4. 异常场景思考
- 数据源不可用怎么办？
- 用户输入错误怎么办？
- 系统故障如何恢复？
- 数据一致性如何保证？

## 5. 合规与安全
- 数据隐私要求（GDPR/等保/行业规范）
- 数据加密要求
- 审计日志要求
- 访问控制要求
```

## 审计提示词模板

```text
# 角色
你是一名资深需求评审专家，擅长发现需求文档中的遗漏、矛盾和不合理之处。

# 任务
请对以下 INITIAL.md 进行严格审计，找出问题并提出修改建议。

# 审计维度

## 1. 完整性检查
- [ ] 是否覆盖了所有用户角色的需求？
- [ ] 是否遗漏了非功能性需求（性能、安全、可用性、可维护性）？
- [ ] 是否明确了系统边界（什么做、什么不做）？
- [ ] 是否定义了成功标准（可量化的验收指标）？
- [ ] 是否考虑了与现有系统的集成需求？

## 2. 一致性检查
- [ ] 功能需求与约束条件是否矛盾？
- [ ] 用户角色定义是否一致（有无遗漏或重复）？
- [ ] 数据来源与功能需求是否匹配？

## 3. 可行性检查
- [ ] 约束条件是否现实可行？
- [ ] 技术选型是否满足约束条件？
- [ ] 是否考虑了部署环境的限制？

## 4. 风险识别
- [ ] 是否存在技术风险？
- [ ] 是否存在合规风险？
- [ ] 是否存在资源风险？

## 5. 优先级检查
- [ ] 是否区分了MVP功能和后续迭代功能？
- [ ] 优先级排序是否合理？

# 输出格式
按标准审计报告格式输出。
```

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | INITIAL.md是否覆盖所有业务需求 | 审计评分≥35/40 |
| 2 | 用户角色定义是否准确 | 无遗漏角色 |
| 3 | 约束条件是否合理可行 | 无矛盾约束 |
| 4 | 系统边界是否清晰 | In Scope/Out of Scope明确 |
| 5 | 风险识别是否充分 | 主要风险已识别并有缓解措施 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **收集信息**：通过提问引导用户补充项目信息
2. **外部搜索**：搜索业界类似系统的最佳实践（至少2个来源）
3. **Brainstorming**：从多个视角进行结构化头脑风暴
4. **生成初稿**：生成 INITIAL.md 初稿
5. **提交审计**：将初稿提交给审计Agent审查
6. **迭代修改**：根据审计报告修改初稿
7. **人类审批**：按审批清单逐项检查
8. **版本化**：将最终版纳入版本管理
9. **移交**：输出作为 Phase 2 的输入

## 注意事项

- **不要跳过搜索**：即使需求看似简单，也要搜索最佳实践
- **不要跳过Brainstorming**：结构化思考避免遗漏
- **MVP优先**：明确区分MVP和后续迭代功能
- **量化指标**：成功标准必须是可量化的
- **系统边界**：明确什么不做，防止范围蔓延
- **风险前置**：早期识别风险，制定缓解措施
