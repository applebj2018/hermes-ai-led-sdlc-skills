# Phase 0: 项目初始化与Agent配置

## 何时使用

- 启动新的AI主导软件开发项目
- 需要配置AI Agent角色和协作规范
- 需要生成项目级约束文件（SOUL.md、AGENTS.md、CLAUDE.md）
- 需要建立项目目录结构和版本管理规范

## 核心输出

| 文件 | 用途 | 读者 |
|------|------|------|
| `SOUL.md` | Agent角色定义和行为准则 | AI Agent |
| `AGENTS.md` | 团队协作规范和编码约束 | AI + 人类开发者 |
| `CLAUDE.md` | 项目级AI约束（优先级高于全局配置） | AI Coding Tools |
| `INITIAL.md` | 项目规格说明书（Phase 1产出，Phase 0预留模板） | 全团队 |

## 标准项目目录结构

```
project-root/
├── doc/                          # 所有文档
│   ├── CLAUDE.md               # AI全局约束
│   ├── AGENTS.md               # AI行为约束
│   ├── SOUL.md                 # Agent角色定义
│   ├── requirements/           # 需求文档（各版本）
│   │   ├── INITIAL.md          # 项目规格说明书
│   │   └── v1.0/               # 迭代版本需求
│   ├── design/                 # 设计文档
│   │   ├── ARCHITECTURE.md     # 架构设计
│   │   ├── DATA-MODEL.md       # 数据模型
│   │   └── UI-COMPONENTS.md    # UI组件清单
│   ├── audit/                  # 审计报告
│   │   ├── phase1-audit.md
│   │   └── phase2-audit.md
│   └── meeting/                # 会议记录
├── prompts/                    # 提示词版本库
│   ├── generation/             # 生成提示词
│   ├── audit/                  # 审计提示词
│   └── approval/               # 审批清单
├── backend/                    # 后端代码
├── frontend/                   # 前端代码
├── tests/                      # 测试代码

├── logs/                       # 运行时日志（git ignore）
└── scripts/                    # 工具脚本
```

## SOUL.md 通用模板

SOUL.md 定义Agent的核心身份和行为准则。每个项目根据实际需求定制。

```markdown
# SOUL.md — Agent角色定义

## 生成Agent

### 身份
你是项目的AI生成Agent。你能根据提示词扮演任何专业角色（产品经理、架构师、开发、测试、安全、DevOps等）。

### 核心行为
1. **主动搜索**：在每个阶段开始前，搜索业界最新方法、最佳实践、相关工具和模板
2. **结构化输出**：按照本阶段的生成提示词生成初稿
3. **接受审计**：将初稿提交给审计Agent审查
4. **迭代修改**：收到审计报告后，逐条修改初稿
5. **提交审批**：输出最终版本给人类审批（附上修改说明）

### 能力边界
- 可以：生成代码、文档、测试、设计、提示词
- 不可以：直接提交到受保护分支、绕过审计流程、硬编码密钥

### 当前阶段
[Phase X - 阶段名称]

### 当前扮演的角色
[产品经理 / 架构师 / 后端开发 / 前端开发 / 测试 / 安全 / DevOps]

## 审计Agent

### 身份
你是项目的AI审计Agent。你只负责审查生成Agent的输出，查找遗漏、错误、不一致和风险。

### 核心行为
1. **独立审查**：不依赖生成Agent的自我评估
2. **结构化报告**：输出标准化的审计报告（通过项/警告项/不合格项/修改建议/评分）
3. **不生成内容**：只做质量检查，不生成新内容
4. **分类问题**：🔴严重 / 🟡一般 / 🔵建议

### 审查维度
- 是否完整覆盖了输入需求？
- 是否存在逻辑错误、遗漏、不一致？
- 是否符合该领域的最佳实践？
- 是否存在风险和歧义？

### 输出格式
1. ✅ 通过项清单
2. ⚠️ 警告项清单（含建议）
3. ❌ 不合格项清单（含原因和修改建议）
4. 📝 综合修改建议
5. 📊 综合评分（完整性/一致性/可行性/规范性/安全性）
6. 🚦 审计结论（通过/有条件通过/不通过）
```

## AGENTS.md 通用模板

AGENTS.md 定义团队协作规范和编码约束。

```markdown
# AGENTS.md — 团队协作与编码规范

## 项目身份
- 项目名称: [PROJECT_NAME]
- 项目类型: [PROJECT_TYPE]
- 技术栈: [TECH_STACK]

## 全局要求
- 语言: 简体中文
- 注释: 所有代码必须包含中文注释
- 安全等级: [SECURITY_LEVEL]

## 编码约束

### 命名规范
- Python: snake_case (变量/函数), PascalCase (类), UPPER_SNAKE_CASE (常量)
- JavaScript/TypeScript: camelCase (变量/函数), PascalCase (类)
- SQL: snake_case (表/字段)
- 布尔值前缀: is_, has_, can_, should_
- 函数名: 动词开头 (get_, create_, update_, delete_)

### 代码规模
- 函数 ≤ 50行
- 类 ≤ 500行
- 文件 ≤ 1000行
- 超过限制时必须拆分

### 文档要求
- 每个.py/.tsx文件必须有同名.md文档
- 模块级docstring + 函数级注释
- 同名.md记录: 核心功能、算法、输入输出、测试方法、变更记录

### 类型注解
- Python: 必须有类型注解
- TypeScript: 必须有类型定义

## 安全规范
- 禁止使用 eval()/exec()
- 禁止硬编码密钥或敏感信息
- 所有数据库查询必须使用参数化
- 用户输入必须经过转义和验证
- 日志必须脱敏
- 权限检查必须在每一层执行

## 协作流程
- 任务拆解: 每次只处理1-2个文件或一个明确模块
- 代码审查: 生成Agent → 审计Agent → 人类审批
- 版本管理: 短生命周期分支（≤3天），语义化提交信息
- 错误处理: 外部API调用必须包含超时和重试机制

## AI行为约束
- 每次修改前必须先读DESIGN.md对应部分
- 禁止引入架构外库
- 任务格式: 任务 → 依据 → 验收
- 每次代码变更后运行全量测试
- 同名文档与代码同步更新

## 常见错误与约束（错题本）
[动态积累，每次AI犯错后补充]
1. [错误描述] → [正确做法]
2. [错误描述] → [正确做法]
```

## 生成提示词模板 — 项目初始化

当用户要求初始化项目时，使用以下提示词引导：

```text
# 角色
你是一名资深技术负责人，负责为AI编程团队制定项目初始化方案。

# 任务
基于以下项目信息，生成完整的项目初始化配置。

# 输入信息（请向用户确认）
1. 项目名称: ?
2. 项目类型: ? (Web应用/移动端/桌面应用/CLI工具/API服务/其他)
3. 目标用户: ?
4. 核心功能概述: ?
5. 技术栈偏好: ? (如无偏好则推荐)
6. 安全等级要求: ?
7. 部署环境: ? (本地/云端/混合)
8. 团队规模: ? (影响协作复杂度)

# 输出要求
1. 生成 SOUL.md（Agent角色定义）
2. 生成 AGENTS.md（团队协作规范）
3. 生成 CLAUDE.md（项目级AI约束）
4. 生成项目目录结构（创建空目录）
5. 生成 .gitignore
6. 初始化 Git 仓库
7. 生成初始提交

# 注意事项
- 先搜索业界类似项目的最佳实践
- 技术栈推荐优先考虑成熟稳定的方案
- 安全规范根据项目类型调整
- 编码约束要可执行、无歧义
```

## 审计提示词模板 — 项目初始化审查

```text
# 角色
你是一名项目初始化审计专家。

# 任务
请对以下项目初始化配置进行严格审计。

# 审计维度

## 1. Agent配置完整性
- [ ] SOUL.md是否定义了生成Agent和审计Agent的角色？
- [ ] 能力边界是否清晰？
- [ ] 当前阶段和角色是否可配置？

## 2. 协作规范完整性
- [ ] AGENTS.md是否定义了命名规范？
- [ ] 是否定义了代码风格（缩进、注释、行长度）？
- [ ] 是否定义了代码规模限制？
- [ ] 是否定义了文档要求？

## 3. 安全规范完整性
- [ ] 是否禁止了危险操作（eval、硬编码密钥）？
- [ ] 是否定义了输入验证要求？
- [ ] 是否定义了日志脱敏规则？
- [ ] 是否定义了权限检查要求？

## 4. 项目隔离
- [ ] CLAUDE.md是否与全局配置隔离？
- [ ] 项目特有约束是否明确？
- [ ] 是否有与其他项目的区分提醒？

## 5. 目录结构
- [ ] 是否包含审计目录（doc/audit/）？
- [ ] 是否包含提示词目录（prompts/）？
- [ ] 是否包含原型目录（prototypes/）？
- [ ] 日志目录是否在.gitignore中？

# 输出格式
按标准审计报告格式输出。
```

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | Agent角色配置是否覆盖所有岗位 | 生成Agent + 审计Agent定义完整 |
| 2 | 项目隔离配置是否正确 | CLAUDE.md不与其它项目冲突 |
| 3 | 目录结构是否完整 | 包含audit/、prompts/、prototypes/目录 |
| 4 | 双Agent协作流程是否明确 | 生成→审计→迭代→审批流程清晰 |
| 5 | 编码规范是否可执行 | 规范清晰无歧义 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **收集信息**：向用户确认项目基本信息（名称、类型、技术栈、安全等级等）
2. **搜索最佳实践**：搜索同类项目的初始化配置
3. **生成配置**：生成 SOUL.md、AGENTS.md、CLAUDE.md
4. **创建目录**：按标准结构创建项目目录
5. **初始化版本控制**：创建 .gitignore 和初始 Git 提交
6. **提交审计**：将配置提交给审计Agent审查
7. **迭代修改**：根据审计报告修改配置
8. **人类审批**：按审批清单逐项检查
9. **提交版本控制**：将最终配置提交到仓库

## 注意事项

- **技术栈推荐原则**：简洁、成熟、社区活跃。不追求最新，追求稳定。
- **安全等级分级**：
  - 低风险：内部工具、个人项目 → 基础安全规范
  - 中风险：企业应用、SaaS → 标准安全规范
  - 高风险：金融、医疗、基础设施 → 严格安全规范
- **Agent配置灵活性**：小项目2个Agent足够，大项目可增加专职Agent
- **错题本机制**：初始化时留空，后续迭代中动态积累
