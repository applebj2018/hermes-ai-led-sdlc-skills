# Phase 3-4: 编码实施

## 何时使用

- 设计文档已批准，进入编码阶段
- 需要生成测试数据
- 需要实现代码（支持TDD或Test-last模式）
- 用户讨论编码、实现、代码生成

## 两种开发模式

### 模式A: TDD（测试驱动开发）

适用场景：需求明确、核心模块、关键业务逻辑

```
Step 1: AI 读取设计文档 → 生成测试用例
Step 2: 审计Agent 审查测试用例
Step 3: AI 根据审计报告补充测试用例
Step 4: 人类审查并补充领域特有场景
Step 5: AI 基于测试用例生成最小实现代码
Step 6: 运行测试 → 不通过 → AI修正代码（不改测试）
Step 7: 测试通过 → 代码审查 + 安全扫描
```

### 模式B: Test-last（先代码后补测试）

适用场景：需求不明确、快速原型、技术探索、UI组件

```
Step 1: AI 读取设计文档 → 生成实现代码
Step 2: 人类验证功能 → 确认行为符合预期
Step 3: AI 基于需求和代码 → 生成测试用例
Step 4: 运行测试 → 发现边界遗漏 → 修正代码或测试
Step 5: 测试通过 → 代码审查 + 安全扫描
```

## 测试数据生成

### 生成提示词模板

```text
# 角色
你是一名资深测试数据工程师。

# 任务
基于设计文档和种子数据，生成全面覆盖的测试数据集。

# 输入
- 设计文档（DESIGN.md, DATA-MODEL.md）
- 种子数据（seed_data/ 目录）

# 输出要求

## 1. 正常场景 (test_data/normal/)
- 覆盖所有数据类型的典型值
- 至少10条记录

## 2. 边界场景 (test_data/boundary/)
- 关键字段的边界值（最小/最大/临界点两侧）
- 空值、null、零值
- 超长字符串、特殊字符

## 3. 异常场景 (test_data/abnormal/)
- SQL注入样本
- XSS样本
- 类型错误
- 格式错误
- 权限越权

## 4. 并发场景 (test_data/concurrent/)
- 并发查询配置
- 并发写入配置

# 输出格式
JSON格式，结构清晰，方便测试代码直接读取。
```

## 代码生成提示词模板

### TDD模式

```text
# 🔴 红阶段：只写失败的测试
"请为[功能]编写测试。该测试应该失败，因为函数/方法还不存在。不要写实现代码。"

# 🟢 绿阶段：只写最少量的代码让测试通过
"现在实现最少的代码来让测试通过。只写足够通过当前测试的代码，不要多写。"

# 🔵 重构阶段：在测试保持通过的前提下优化代码
"重构实现以提升代码质量。重构后测试必须保持绿色。"
```

### Test-last模式

```text
# 角色
你是一名资深软件工程师。

# 任务
基于以下设计文档，生成[模块/函数]的实现代码。

# 输入
1. 设计文档片段（算法、输入输出、边界条件）
2. 架构约束（技术栈、代码风格、禁止项）

# 输出要求
1. 实现代码：包含完整的模块级docstring和函数级注释
2. 同名文档：记录核心功能、算法、输入输出、测试方法
3. 单元测试：覆盖正常、边界、异常路径

# 约束
- 遵循AGENTS.md中的编码规范
- 函数 ≤ 50行，类 ≤ 500行
- 必须有类型注解和中文注释
- 不要实现设计文档未覆盖的功能
```

## 迭代开发原则

### 小步快跑

每次只处理 **1-2个文件或一个明确模块**：

```
❌ 错误：一次性要求AI生成整个模块
"请实现整个系统的后端代码，包括所有功能"

✅ 正确：拆解为多个小任务
任务1: 实现文档上传的基础API端点
任务2: 为上传API添加文件类型校验
任务3: 实现上传后的元数据提取
任务4: 集成对象存储
```

### 渐进式增强

```
第1轮: 实现核心功能的Happy Path → 测试通过
第2轮: 添加输入验证和错误处理 → 测试通过
第3轮: 添加性能优化和缓存 → 测试通过
第4轮: 添加日志和可观测性 → 测试通过
```

### UI组件渐进迭代

```
第1轮: 生成静态布局 + 样式
第2轮: 添加Props定义和类型
第3轮: 添加交互逻辑（事件处理）
第4轮: 集成API调用和数据绑定
第5轮: 优化性能（memo、懒加载）
```

## 代码审查提示词模板

```text
# 角色
你是一名代码审计专家。

# 审计维度

## 1. 安全性
- [ ] 是否存在SQL注入风险？
- [ ] 是否存在XSS风险？
- [ ] 是否硬编码了密钥或敏感信息？
- [ ] 是否使用了eval()/exec()等危险函数？
- [ ] 权限检查是否在每一层执行？

## 2. 逻辑正确性
- [ ] 是否覆盖了设计文档中定义的边界条件？
- [ ] 异常处理是否完整？
- [ ] 返回值类型是否与文档定义一致？
- [ ] 是否有内存泄漏或资源未释放？

## 3. 规范性
- [ ] 是否遵循AGENTS.md中的编码规范？
- [ ] 函数长度是否≤50行？
- [ ] 类长度是否≤500行？
- [ ] 是否有类型注解？
- [ ] 是否有中文注释？
- [ ] 同名.md文档是否已同步更新？

## 4. 测试覆盖
- [ ] 是否覆盖了正常场景？
- [ ] 是否覆盖了边界场景？
- [ ] 是否覆盖了异常场景？

## 5. 架构一致性
- [ ] 是否引入了架构外的库？
- [ ] 模块职责是否符合架构设计？
- [ ] 接口定义是否与设计文档一致？

# 输出格式
按标准审计报告格式输出。
```

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | 代码是否遵循编码规范 | 审计报告无❌项 |
| 2 | 安全漏洞是否已修复 | 无SQL注入/XSS风险 |
| 3 | 测试覆盖度是否达标 | 覆盖正常/边界/异常场景 |
| 4 | 同名文档是否同步更新 | 文档与代码一致 |
| 5 | 架构一致性是否保持 | 无架构外依赖 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **确认上游**：确保设计文档和测试数据已完整
2. **任务拆解**：将大功能拆解为2-5分钟可完成的小任务
3. **选择模式**：根据场景选择TDD或Test-last模式
4. **生成代码**：按小任务逐个生成代码
5. **运行测试**：每完成一个小任务立即运行测试
6. **代码审查**：提交给审计Agent审查
7. **迭代修改**：根据审计报告修改代码
8. **人类审批**：按审批清单逐项检查
9. **版本提交**：将代码、文档、测试一起提交

## 注意事项

- **Checkpoint保护**：每次AI修改前创建Git checkpoint
- **改提示词而非改代码**：提示词是源，代码是输出
- **AI可能偷懒**：必须审查代码，AI可能遗漏功能
- **每次变更后运行测试**：确保未破坏已有功能
- **文档同步**：代码和同名.md必须同步更新
- **错题积累**：反复出现的错误写入AGENTS.md错题本
