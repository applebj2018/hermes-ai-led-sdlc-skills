# Phase 6: 文档同步与交付

## 何时使用

- 代码和测试已完成，需要生成交付文档
- 需要同步代码与文档
- 需要生成交付指南、用户手册、变更日志
- 用户讨论文档、交付、手册

## 核心原则

### HARD-GATE（阶段入口/出口）

```
< HARD-GATE: Phase 6 入口 >
- 前置检查：确认代码已通过测试和审计
- 前置检查：确认模块级同名 .md 文档已随代码同步更新
- 未完成的测试不得进入文档交付阶段
</ HARD-GATE >

< HARD-GATE: Phase 6 出口 >
- 禁止跳过文档审计：交付文档必须经 AI 审计完整性和准确性
- 禁止跳过人类审批：交付文档必须经用户确认
- 禁止文档不一致：代码与文档不一致时，必须同步修改
</ HARD-GATE >
```

### Anti-Pattern（常见错误模式）

**"文档是最后的事，先交付再说"** — 文档是项目的记忆。没有文档的项目，维护者无法理解设计意图，每次修改都是盲猜。

**"文档照抄代码"** — 文档不是代码的翻译。文档解释为什么（why），代码实现做什么（what）。两者互补，不重复。

**"CHANGELOG 随便写"** — 变更日志是用户和运维的沟通桥梁。不准确的变更日志导致部署错误和回滚。

### 强制文档同步

| 代码变更 | 必须同步的文档 | 同步方向 |
|---------|--------------|---------|
| 修改核心类/函数 | 对应的 module.md + DESIGN.md | 代码→文档 |
| 架构组件变更 | ARCHITECTURE.md | 代码→文档 |
| 引入新依赖 | INITIAL.md | 代码→文档 |
| UI组件变更 | UI-COMPONENTS.md | 代码→文档 |
| 修改权限逻辑 | PERMISSION-DESIGN.md | 代码→文档 |
| 修改审批流程 | WORKFLOW-DESIGN.md | 代码→文档 |
| 修改版本管理 | VERSION-DESIGN.md | 代码→文档 |
| 任何功能变更 | CHANGELOG.md | 代码→文档 |

**黄金法则**：不一致时，确保代码和文档一致——代码是事实，文档是描述。两者同步改。

## 文档生成提示词

```text
# 角色
你是一名技术文档专家，擅长编写清晰、完整的技术文档。

# 任务
基于项目的所有设计文档和代码，生成交付文档。

**请先搜索互联网**，参考业界交付文档的最佳实践。

# 输入
- 所有设计文档（INITIAL.md, ARCHITECTURE.md, DESIGN.md, ...）
- 代码仓库
- 测试报告

# 输出要求
1. DELIVERY-GUIDE.md — 面向运维和客户的交付指南
2. USER-GUIDE.md — 面向最终用户的操作手册
3. CHANGELOG.md — 版本变更记录

# 文档要求
- 语言清晰易懂
- 包含足够的示例
- 操作步骤可执行
- 与代码实现一致
```

## 文档审计提示词

```text
# 角色
你是一名文档审计专家。

# 审计维度

## 1. 完整性
- [ ] 是否覆盖了所有功能模块？
- [ ] 是否包含了部署和运维的完整步骤？
- [ ] 是否有故障排查指南？

## 2. 准确性
- [ ] 文档描述是否与代码实现一致？
- [ ] 配置参数是否正确？
- [ ] 操作步骤是否可执行？

## 3. 可读性
- [ ] 语言是否清晰易懂？
- [ ] 是否有足够的示例？
- [ ] 是否有截图或流程图辅助说明？

## 4. 一致性
- [ ] 文档风格是否统一？
- [ ] 术语是否一致？
- [ ] 版本号是否与代码一致？

# 输出格式
按标准审计报告格式输出。
```

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | 交付文档是否完整 | 所有交付物齐全 |
| 2 | 文档与代码是否一致 | 无不一致项 |
| 3 | 用户手册是否可操作 | 步骤可执行 |
| 4 | 运维指南是否完整 | 部署步骤完整 |
| 5 | CHANGELOG是否准确 | 变更记录完整 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **检查文档同步**：确认每个模块有对应的同名.md文档
2. **生成交付文档**：生成 DELIVERY-GUIDE.md 和 USER-GUIDE.md
3. **更新CHANGELOG**：按语义化版本记录变更
4. **提交审计**：将文档提交给审计Agent审查
5. **迭代修改**：根据审计报告修正文档
6. **人类审批**：按审批清单逐项检查
7. **版本化**：将所有文档提交到版本管理

## 注意事项

- **文档是代码的锚点**：AI改代码前先读对应.md文件
- **同步更新**：代码变更必须同步更新文档
- **可操作性**：用户手册的步骤必须可执行
- **一致性检查**：定期抽查代码与文档的一致性
- **版本对应**：文档版本号与代码版本号一致
- **培训材料编写**：编写技术培训文档时，参考 `references/training-material-pattern.md`（例子先行、能力阶段、动手实践导向）
