# Phase 8: 版本管理与发布

## 何时使用

- 需要管理代码分支和版本
- 需要发布补丁或新版本
- 需要执行回滚操作
- 用户讨论版本、发布、补丁、回滚

## 分支策略

### 推荐：主干开发模型

```
main ──────────────────────────────────────●
     \                                    /
      ├── branches/feat/multi-doc-agg ── PR
      ├── branches/fix/permission-bug ── PR
      └── branches/refactor/search-opt ─ PR
```

### 分支命名规范

```
type/description

type:
  - feat/    新功能
  - fix/     Bug修复
  - refactor/ 重构
  - chore/   构建/工具变更
  - docs/    文档变更
  - test/    测试变更
```

### 分支规则

1. **AI不直接提交到主干**：必须通过PR/MR合并
2. **短生命周期**：分支存活不超过3天
3. **Feature Flag**：未完成功能用Feature Flag控制
4. **强制审查**：所有AI生成的代码必须有人工审查

## 提交信息规范

### 格式

```
type(scope): description

[optional body]
```

### 类型

| Type | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档变更 |
| style | 代码格式（不影响代码含义） |
| refactor | 代码重构 |
| test | 测试相关 |
| chore | 构建/工具/依赖变更 |

### 示例

```
feat(search): 添加多文档片段聚合检索功能
fix(permission): 修复越权访问机密文档的问题
docs(api): 更新API文档
refactor(auth): 重构认证模块
```

## 语义化版本

### 版本号格式

```
MAJOR.MINOR.PATCH

MAJOR: 不兼容的API变更
MINOR: 向后兼容的功能新增
PATCH: 向后兼容的Bug修复
```

### 版本发布流程

```
1. 确认所有测试通过
2. 更新CHANGELOG.md
3. 打标签: git tag v1.2.3
4. 推送标签: git push origin v1.2.3
5. 生成发布说明
6. 执行部署
7. 验证部署
```

## 补丁修订流程

```
1. 安全漏洞/紧急Bug报告
   ↓
2. AI分析影响范围 (git blame + 依赖分析)
   ↓
3. AI生成补丁代码 + 补丁测试
   ↓
4. 审计Agent审查补丁
   ↓
5. 人工审查补丁
   ↓
6. 全量冒烟测试 (< 2分钟)
   ↓
7. 合并到release分支
   ↓
8. 自动触发部署流水线
```

## 补丁审查提示词

```text
# 角色
你是一名补丁审计专家。

# 审计维度

## 1. 修复有效性
- [ ] 补丁是否真正修复了问题？
- [ ] 是否有回归测试验证修复？

## 2. 副作用分析
- [ ] 补丁是否影响了其他功能？
- [ ] TDAD映射中的受影响测试是否全部通过？

## 3. 安全性
- [ ] 补丁是否引入了新的安全漏洞？
- [ ] 是否有敏感信息泄露？

## 4. 规范性
- [ ] 补丁代码是否遵循编码规范？
- [ ] 是否有对应的文档更新？

# 输出格式
按标准审计报告格式输出。
```

## 回滚机制

### 回滚触发条件

- 发布后出现严重Bug
- 性能显著下降
- 安全漏洞
- 数据不一致

### 回滚步骤

```
1. 确认问题严重性
2. 决定回滚版本
3. 执行回滚: git revert <commit> 或 git reset --hard <tag>
4. 重新部署
5. 验证回滚
6. 记录回滚原因
7. 分析根本原因
8. 制定修复计划
```

## 人类审批清单

| # | 审批项 | 通过标准 |
|---|--------|---------|
| 1 | 补丁是否有效修复问题 | 回归测试通过 |
| 2 | 补丁是否有副作用 | TDAD测试全部通过 |
| 3 | 补丁安全性是否达标 | 无新漏洞 |
| 4 | 部署流程是否执行 | 部署成功 |
| 5 | 回滚方案是否可行 | 有明确的回滚步骤 |
| 6 | 审计报告中的问题是否已修复 | 所有❌项已解决 |

## 执行步骤

1. **分支管理**：按规范创建和管理分支
2. **代码审查**：所有PR必须经过审查
3. **测试验证**：合并前运行全量测试
4. **版本打标签**：按语义化版本打标签
5. **生成发布说明**：更新CHANGELOG.md
6. **执行部署**：按部署指南执行
7. **验证部署**：冒烟测试确认
8. **监控**：部署后持续监控

## 注意事项

- **短分支**：分支存活不超过3天，避免合并冲突
- **小提交**：每次提交只包含一个逻辑变更
- **原子提交**：提交要么全部成功，要么全部回滚
- **标签保护**：生产环境标签不允许删除
- **回滚准备**：每次发布前准备好回滚方案
- **监控告警**：部署后持续监控，发现问题立即回滚
